# 仕様書駆動 + TDD ワークフロー(PlanDiff)

このリポジトリでの実装作業は、必ず以下の順序で進める。順序の省略・入れ替えは禁止。

## ワークフロー

### 0. 指示資料を確認する(タスク着手時・必須)

- `docs/指示資料/` を確認し、今回のタスクに関連する資料(メモ・画面ラフ・参考記事等)がないか調べる
- 関連資料があれば内容を仕様書に反映し、仕様書の「関連」欄に資料パスを記載する
- 資料と要件定義書・CLAUDE.md の内容が矛盾する場合は、**勝手に解釈せずユーザーに確認する**

### 1. 仕様書を作成する(実装前・必須)

- `docs/specs/<機能名>.md` を作成する。テンプレート:

```markdown
# 仕様書: <機能名>

- ステータス: ドラフト | 承認済み(承認日)
- 関連: docs/要件定義書.md の FR-XX / docs/指示資料/<参照した資料>(あれば)

## 目的
## 仕様(入出力・画面・データ変更・エラー時の挙動)
## スコープ外
## テストシナリオ
<!-- Given / When / Then 形式で列挙。正常系・異常系・境界値を含め、
     各シナリオに [単体] / [結合] のレベルを付ける -->
- S1 [単体]: Given ... When ... Then ...
- S2 [結合]: Given ... When ... Then ...
```

- **網羅性の基準**: 正常系・異常系・境界値のそれぞれに最低1シナリオ。結合レベル(Route Handler / Server Action〜DB、コンポーネント間連携)のシナリオを必ず含める。結合シナリオが不要と判断した場合は、その理由を仕様書に明記してユーザー承認を得る

### 2. ユーザーの承認を得る

- 仕様書を提示し、**明示的な承認を得るまで実装コードを書かない**
- 承認されたら仕様書のステータスを「承認済み」に更新する

### 3. テストシナリオからテストを展開する(Red)

- 仕様書の `## テストシナリオ` の各項目を `tests/` 配下のテストケースに1対1で対応させる
- テスト名にシナリオID(S1等)を含め、仕様書と突き合わせ可能にする
- まずテストが**失敗することを確認**してから実装に進む

### 4. 実装する(Green → Refactor)

- テストを通す最小の実装 → リファクタリング
- **禁止**: `it.skip` / `describe.skip` / `it.todo` / テストのコメントアウト / `--passWithNoTests` によって「全件通過」と見なすこと。通らないテストがあるなら、実装を直すか、仕様変更としてユーザーに確認する

### 5. 検証する

- `npm run check`(typecheck + lint + test + build)を実行し、**実際の出力を確認してから**完了を報告する
- **完了条件**: 仕様書の全シナリオ(単体・結合)に対応するテストが存在し、**全件合格**していること。テスト件数がシナリオ数を下回っていないかを突き合わせて確認する

### 6. 行動履歴を記録する

- `docs/logs/YYYY-MM-DD.md` に追記(なければ作成)。形式:

```markdown
## HH:MM <作業タイトル>
- やったこと:
- 理由:
- 結果(テスト・ビルドの出力要約):
- 未解決・次にやること:
```

## 例外処理

- **不明点が出たら**: 推測で進めず、その場でユーザーに確認する(AskUserQuestion)
- **同じバグを3回出したら**: `CLAUDE.md` の「バグ由来ルール」セクションに再発防止ルールを追記し、ログに記録する。バグの発生回数はログから判断する
