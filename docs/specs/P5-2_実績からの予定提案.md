## 目的

過去の計測実績から「繰り返しやっている作業」を検出し、予定として提案することで、
計画作成の手間を減らす。記録はしているが計画に載っていない作業を計画に取り込むことで、
計画と実績のギャップ(FR-06/FR-07)をより実態に近づける。

### 決定済みの要件(brainstorming結果)

| 論点 | 決定 |
|---|---|
| コア体験 | 実績からの予定化(丸ごと提案)。所要時間サジェストや係数可視化は後続 |
| 提案の場所 | カレンダー画面に「実績からの提案」セクションを常設 |
| 検出の軸 | 同タイトル × 同曜日 × 近い時間帯の繰り返し実績を日時入りで提案 |
| 受け入れ形態 | 「この週だけ(単発予定)」/「毎週(P5-1の定期予定)」をユーザーが選択 |
| 却下 | その場限りの非表示(クライアントstateのみ。永続化しない) |
| 実現方式 | 案A: 純粋関数+既存Server Action再利用。**DB変更なし** |

## 仕様

### アーキテクチャ概要(案A)

```
カレンダーpage (Server Component)
  ├─ fetchSuggestionSourceEntries(表示週開始前4週の完了実績)  ← 新設(lib/timer/entries.ts)
  ├─ fetchSyncedEvents(表示週±1週)                            ← 既存
  ├─ fetchRecurringRules()                                     ← 既存
  └─ props で CalendarView 経由 → PlanSuggestions (client) へ
        └─ computeSuggestions(純粋関数。ブラウザのIANAタイムゾーンで判定)
              ├─ この週に追加 → createAppEventAction(既存)
              ├─ 毎週にする   → createRecurringRuleAction(既存)
              └─ ×(非表示)  → useState のみ
```

- 曜日・時刻の判定はユーザーのローカルタイムゾーンに依存するため、`computeSuggestions` は
  クライアント側で `Intl.DateTimeFormat().resolvedOptions().timeZone` を渡して実行する
  (P5-1のルール作成・「表示上の選択日はクライアントが確定」と同じ方針)。
  純粋関数なのでテストは単体で網羅できる
- サーバー側の取得範囲はタイムゾーン差を吸収するバッファ込みの概算とし、
  正確な週境界のフィルタは純粋関数側で行う(P1-2の同期範囲と同じ考え方)

### データ取得(`lib/timer/entries.ts` に追加)

- `fetchSuggestionSourceEntries(client, baseDate)`: 完了済み実績(`end_at IS NOT NULL`)を
  「基準日を含む週の開始 − 4週間 − TZバッファ1日」〜「週の開始 + TZバッファ1日」の範囲で取得
  (検出に使うのは表示週開始より前の実績のみ。正確な週境界は純粋関数側でローカルTZ判定)。
  返却形は既存 `TimeEntryItem` と同一(id / title / googleEventId / startAt / endAt)
- RLSにより本人の行のみ。取得失敗時は例外を投げず空配列を返し、提案セクションを出さないだけに留める
  (カレンダー本体の描画を妨げない)

### 検出ロジック(`lib/calendar/suggestions.ts` の純粋関数 `computeSuggestions`)

入力: 実績リスト、表示週の予定(`SyncedEvent[]`)、定期ルール一覧(`RecurringRuleSummary[]`)、
表示週の基準日、現在時刻、IANAタイムゾーン。出力: 提案リスト(最大3件)。

1. **前提条件**: 表示週の終端が現在時刻より前(=過去週を表示中)なら空を返す
2. **対象実績**: 表示週開始(ローカルTZ)より前の**28日間**に開始された完了実績。
   タイトルはtrimして同一視し、空タイトルは除外する
3. **クラスタリング**: タイトル × 曜日(ローカルTZ)でグループ化し、グループ内で
   ローカル開始時刻(0:00からの分)の差が**60分以内**に収まる最大の集合をクラスタとする。
   クラスタ内の**発生日数が2日以上**なら提案候補(同じ日の重複計測は1日と数える)
4. **提案日時**:
   - 開始時刻 = クラスタのローカル開始時刻の中央値を**15分単位に丸め**(四捨五入)
   - 所要時間 = クラスタの実績時間の中央値を**15分単位に丸め**(四捨五入、最低15分)
   - 提案日 = 表示週の該当曜日。終了 = 開始 + 所要時間
5. **除外条件**(該当したら候補から外す):
   - 表示週の同じ曜日に、同タイトル(trim一致)の予定(`synced_events`)が既にある
   - 同タイトルの定期ルールが存在し、そのパターンが該当曜日を含む
     (daily=全曜日、weekdays=月〜金、weekly=weekdays配列で判定)
   - 提案の開始日時が現在時刻より前(今週表示なら過ぎた曜日には提案しない。未来週は全曜日対象)
   - 丸め後の終了時刻が日をまたぐ(24:00を超える)候補
     (FR-12の定期予定は日またぎ不可のため「毎週にする」が成立しない。単発でも提案しない)
6. **順序と上限**: 発生日数の降順(同数なら提案日時の昇順)で**最大3件**

提案1件のデータ形(クライアント内のみ。DB保存しない):

```ts
interface PlanSuggestion {
  key: string;            // "タイトル|曜日" 。セッション内の非表示管理に使う
  title: string;
  weekday: number;        // 0=日〜6=土(JS Date#getDay と一致)
  date: string;           // 提案日 "YYYY-MM-DD"(表示週の該当日、ローカルTZ)
  startTime: string;      // "HH:mm"(ローカルTZ)
  endTime: string;        // "HH:mm"(ローカルTZ)
  occurrenceCount: number; // 直近4週での発生日数
}
```

### 受け入れ・却下(`components/plan-suggestions.tsx`)

- **この週に追加**: `date + startTime/endTime`(ローカルTZ)をUTC ISOへ変換し、
  既存 `createAppEventAction({ title, startAt, endAt })` を呼ぶ。成功時はカードを消し
  `router.refresh()`(除外条件により再計算後も出ない)
- **毎週にする**: 既存 `createRecurringRuleAction` を
  `{ title, pattern: "weekly", weekdays: [weekday], startTime, endTime, timezone, startsOn: date, endsOn: null }`
  で呼ぶ。成功時は同上
- **×(非表示)**: `useState` で当該 `key` を非表示にするだけ。Action呼び出し・永続化なし。
  リロードや再訪問で再び表示されてよい(brainstormingで確定)
- 実行中の多重タップは処理中disabledで防ぐ

### UI(実装時は ui-quality Skill・375px確認)

- カレンダー画面のタイムライン上部に「実績からの提案」セクション。
  **候補0件・過去週表示・データ取得失敗時はセクションごと非表示**(空枠を出さない)
- カード1枚の構成:
  - タイトル(実績のタイトルそのまま)
  - 説明文: 「毎週◯曜 HH:MM頃・約N分(直近4週でK回)」
    (Nは所要分。60分以上は「約1時間30分」等の既存表記があればそれに合わせる)
  - 操作: 「この週に追加」「毎週にする」「×」
- 文言は `lib/calendar/messages.ts` に集約(日本語)
- 375pxで操作ボタンが折り返しても押しやすいこと。デザインはD-1トークン
  (群青×柿パレット)に従う。予定側の提案なので「予定=計画」系の色(群青)を基調とする

### エラー時の挙動一覧(すべて日本語。`lib/calendar/messages.ts` に集約)

| 状況 | 挙動 |
|---|---|
| 過去実績の取得失敗(サーバー) | 提案セクションを表示しない。カレンダー本体は通常どおり描画 |
| 「この週に追加」失敗 | カード内にエラーメッセージ表示。カードは残り再試行できる |
| 「毎週にする」失敗 | 同上 |
| 未ログイン(Action到達時) | 既存Actionの挙動どおり `/login` へリダイレクト |

### セキュリティ

- 新しいデータ経路なし。読み取りは既存RLS(本人の行のみ)、書き込みは既存Server Action
  (認証確認済み)を再利用する
- 提案はクライアント計算のみでDB保存しないため、新規テーブル・ポリシー追加なし

## スコープ外

- 却下の永続化(「二度と提案しない」)
- 見積もり係数(予定vs実績の倍率)の学習・表示・補正提案(v2の残り)
- 予定作成パネル内の所要時間サジェスト(将来の拡張)
- 時間帯不問の頻出タイトル提案
- 「平日毎日」等、週1回パターン以外の定期化提案(受け入れは weekly 単一曜日のみ。
  複数曜日にまたがるパターンは曜日ごとに別カードとして扱う)
- サマリー画面・計測画面への提案表示

### 手動疎通確認(自動テストの補完。DoDに含める)

- M1: 375pxで、繰り返し実績のあるユーザーで `/calendar` を開く → 提案カードが表示される →
  「この週に追加」→ カレンダーに予定が現れ、タイマー開始できる

  同じ候補が提案に再表示されない

  ダークモードで375pxの表示崩れがない
