# 仕様書: P4-1 オンボーディング

- ステータス: 承認済み(承認日 2026-07-11)
- 関連: docs/要件定義書.md §7(画面「オンボーディング」)/ docs/開発計画.md P4-1 / docs/指示資料/(該当資料なし。README のみ)

## 目的

初回ログインしたユーザーに「PlanDiff の使い方 3 ステップ」を提示し、
コアループ(予定を立てる → タイマーで記録 → ギャップを見る)へ最短で誘導する。
成功指標「予定連動タイマーを 1 回以上使った割合 ≥ 50%」(要件定義書 §13)の起点となる画面。

**要件定義書からの変更点**: 要件定義書 §7 では「カレンダー連携の許可 + 3 ステップ説明」だが、
Google 連携は P2-5 で UI 凍結中(`GOOGLE_INTEGRATION_ENABLED` 既定 OFF)のため、
本仕様では **連携許可ステップを含めず**、アプリ内予定を前提とした 3 ステップ説明のみとする。
凍結解除時の連携許可ステップ追加は本仕様のスコープ外(下記参照)。

## 仕様

### データ変更(マイグレーション → RLS → アプリコードの順)

`profiles` に既読管理カラムを 1 列追加する:

```sql
alter table public.profiles
  add column onboarded_at timestamptz;
```

- `NULL` = オンボーディング未完了(初回表示対象)。値あり = 完了日時(UTC)
- 新規 RLS ポリシーは不要(既存の `profiles_select_own` / `profiles_update_own` が本人のみの読み書きをすでに保証している)
- 既存ユーザーは `NULL` のままとする(= 次回 `/calendar` 訪問時に 1 回だけオンボーディングが表示される)。現時点の登録ユーザーは開発者のみのため許容する
- ローカル検証: `npx supabase db reset` → 既存テスト全件合格 → `npx supabase db diff` で意図しない差分がないこと
- 本番への `db push` は従来どおりユーザー実行(P2-5 と同じ運用)

### 画面: `/onboarding`(`app/(app)/onboarding/page.tsx`)

認証必須グループ配下(未ログインは既存の `(app)/layout.tsx` が `/login` へリダイレクト)。

3 枚のステップカードをステップインジケーター付きで表示する(クライアントコンポーネント。
ステップ切替のインタラクションのみに限定し、ページ自体は Server Component から呼び出す):

| ステップ | タイトル | 説明(要旨) |
|---|---|---|
| 1 | 予定を立てる | カレンダーの「+」から今日の予定を作成する。まずは 1 つで OK |
| 2 | タイマーで記録する | 予定をタップしてタイマー開始。割り込み作業はフリータイマーで記録 |
| 3 | ギャップを見る | 予定(左・薄)と実績(右・濃)の重なりでズレが見える。サマリーで今日/今週の集計も確認 |

- ナビゲーション: 「次へ」/「戻る」(ステップ 1 では「戻る」を出さない)。最終ステップは「次へ」の代わりに「はじめる」
- ヘッダーに「スキップ」リンクを常時表示
- 「はじめる」「スキップ」のどちらも Server Action `completeOnboardingAction` を実行し、`profiles.onboarded_at = now()` を保存して `/calendar` へリダイレクトする
- 完了済みユーザーが直接 `/onboarding` を開いた場合もそのまま閲覧できる(再閲覧用。`onboarded_at` は上書きしない挙動でよい ※ `now()` で更新されても実害はないため実装は同一 Action で構わない)
- 文言はすべて日本語。`lib/onboarding/messages.ts` に寄せる(既存の messages パターンに合わせる)
- モバイルファースト(375px 基準)。ダークモード対応。イラスト・画像は使わずテキスト+既存トーンの装飾で構成する(MVP)

### 導線

1. **初回リダイレクト**: `/calendar`(ログイン後の着地点)で `profiles` 取得時に `onboarded_at` も select し、`NULL` なら `redirect("/onboarding")`。`/summary` `/settings` はゲートしない(オンボーディングは強制導線を最小にする)
2. **再閲覧**: 設定画面(`/settings`)に「使い方をもう一度見る」リンク → `/onboarding`

### Server Action: `completeOnboardingAction`

- 配置: `app/(app)/onboarding/actions.ts`
- 処理: 認証ユーザーの `profiles.onboarded_at` を `now()` に更新(authenticated クライアント経由 = RLS で本人のみ)→ `redirect("/calendar")`
- エラー時: 日本語エラーメッセージをオンボーディング画面に表示し、画面に留まる(握りつぶさない)

### エラー時の挙動

- `completeOnboardingAction` の DB 更新失敗: 「保存に失敗しました。もう一度お試しください。」を表示し、`/calendar` へは遷移しない
- `/calendar` での `profiles` 取得失敗: リダイレクト判定は行わず従来どおりカレンダーを表示する(オンボーディングのためにコア画面を止めない)

## スコープ外

- Google 連携許可ステップ(凍結解除時に本仕様書を改定して追加を検討)
- イラスト・アニメーション等のリッチ表現(P4-5 磨き込みで再検討)
- オンボーディング内でのサンプル予定の自動作成(v2 検討)
- チュートリアルのステップ内インタラクション(実画面ツアー等)

## テストシナリオ

正常系:

- S1 [単体]: Given オンボーディング画面のステップ 1 When 表示 Then タイトル・説明・「次へ」・「スキップ」が表示され「戻る」は表示されない
- S2 [単体]: Given ステップ 1 When 「次へ」を 2 回押す Then ステップ 3 が表示され「はじめる」ボタンに変わる。「戻る」でステップ 2 に戻れる
- S3 [単体]: Given ステップ 3 When 「はじめる」を押す Then `completeOnboardingAction` が呼ばれる
- S4 [単体]: Given 任意のステップ When 「スキップ」を押す Then `completeOnboardingAction` が呼ばれる
- S5 [結合]: Given `onboarded_at` が NULL の認証ユーザー When `completeOnboardingAction` を実行 Then `profiles.onboarded_at` に現在時刻(UTC)が保存され `/calendar` へリダイレクトされる
- S6 [結合]: Given `onboarded_at` が NULL のユーザー When `/calendar` を表示 Then `/onboarding` へリダイレクトされる
- S7 [結合]: Given `onboarded_at` が設定済みのユーザー When `/calendar` を表示 Then リダイレクトされずカレンダーが表示される

異常系:

- S8 [単体]: Given ステップ 3 When `completeOnboardingAction` が失敗 Then 日本語エラーメッセージが表示され画面に留まる
- S9 [結合]: Given 認証ユーザー A When A のクライアント(RLS)で他ユーザー B の `profiles.onboarded_at` を更新 Then 更新されない(0 行更新)

境界値:

- S10 [結合]: Given `onboarded_at` 設定済みのユーザー When `/onboarding` を直接開く Then リダイレクトされず閲覧できる(再閲覧導線)
- S11 [結合]: Given マイグレーション適用後 When 既存ユーザーの `profiles` を select Then `onboarded_at` は NULL(既存データを壊さない)

手動確認(自動テスト対象外):

- M1: 375px 幅で全 3 ステップ・スキップ・「はじめる」→カレンダー遷移を確認(ライト/ダーク)
- M2: 新規ユーザーでサインアップ → メール確認 → 初回 `/calendar` でオンボーディングが表示され、完了後は再表示されないこと
