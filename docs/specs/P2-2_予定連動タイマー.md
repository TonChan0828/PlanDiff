- ステータス: 承認済み(2026-07-07。実績最小表示=P2-2に含める、タップで即開始/停止で確定)
- 関連: docs/要件定義書.md の FR-04 / §5(FR-04/05 タイマー)/ §6(カレンダー=メイン画面)、docs/開発計画.md P2-2
- 依存: P2-1(カレンダービュー)、P0-6(`time_entries` テーブルと partial unique index)
- 指示資料: なし(docs/指示資料/ を確認済み、2026-07-07時点でREADMEのみ)

## 目的

カレンダービューの予定ブロックをタップして、その予定に紐づくタイマーを開始/停止できるようにする。
実行中タイマーは常に1本(サーバー側=DB制約で保証)。新規開始時は既存タイマーを自動停止する。
タイマー状態はDBに永続化し、リロード・複数デバイスをまたいで復元できる。

## 仕様

### 全体構成

```
lib/timer/service.ts                 … タイマー操作のコアロジック(SupabaseClientを受け取る。結合テスト対象)
lib/timer/entries.ts                 … time_entries 読み取りヘルパー(実行中エントリ+表示範囲の実績)
lib/timer/elapsed.ts                 … 経過時間の整形(表示用)
lib/timer/messages.ts                … タイマーUI文言定数
app/(app)/calendar/timer-actions.ts  … Server Actions(認証確認 → service呼び出し → revalidate)
components/running-timer-bar.tsx     … 実行中タイマーバー(クライアント)
components/calendar-view.tsx         … 予定ブロックのボタン化+実績レーン表示(変更)
app/(app)/calendar/page.tsx          … 実行中エントリ+表示範囲の実績の取得を追加(変更)
```

- Google APIは一切使わない(`synced_events` キャッシュ由来のデータのみ)。DB書き込みはServer Action経由のサーバーのみ
- タイマーはリアルタイム性が要るため、CLAUDE.mdの原則どおり**クライアント操作+楽観的更新**とし、
  初期状態(実行中エントリ・実績)はServer Component(page)→ propsで渡す

### 操作(予定ブロックタップ)

| 状態 | タップした予定 | 挙動 |
|---|---|---|
| 実行中なし | 任意の予定 | その予定のタイマーを開始 |
| その予定のタイマーが実行中 | 同じ予定 | 停止(実績として確定) |
| 別のタイマーが実行中 | 別の予定 | 既存を自動停止し、新規開始(**エラーにしない**) |

- 予定ブロックは `<li>` から `<button>` 化する(P2-1で構造を想定済み)。実行中の予定ブロックは枠線強調+「記録中」表示で判別できるようにする(色だけに頼らない)
- 過去・未来の日の予定でも開始できる(開始時刻は常に「現在」。予定の時間帯とは無関係)
- 週ビューでも同様にタップで開始/停止できる

### Server Action / データ変更

**startTimer(googleEventId, title)**

1. 認証確認(未認証は `/login` へ)
2. 実行中エントリ(`end_at IS NULL`)があれば `end_at = now` で停止
3. `time_entries` に insert: `title` = 予定タイトルのスナップショット、`google_event_id` = 予定のID、`start_at` = サーバー現在時刻、`end_at` = null
4. 競合(partial unique index違反 = 別デバイスが同時に開始)時は、停止→insertを**1回だけリトライ**。それでも失敗したらエラーを返す

**stopTimer()**

1. 認証確認
2. 実行中エントリの `end_at = now` を設定して実績として確定
3. 実行中エントリがない場合(別デバイスで停止済み等)は**何もせず成功扱い**(冪等)

- 時刻はすべてサーバー側で決定する(クライアントの時計に依存しない)。DB保存はUTC
- 経過1分未満で停止した0分〜短時間エントリもそのまま保存する(削除・修正はP2-4の手動編集で対応)
- スキーマ変更・マイグレーション: **なし**(P0-6の `time_entries` と partial unique index をそのまま使う)

### 実行中タイマーバー

- 実行中エントリがあるときだけ、カレンダー画面の下部に固定(sticky)表示する
- 内容: タイトル(なければ「(タイトルなし)」)+経過時間(`H:MM:SS`)+「停止」ボタン(タップターゲット44px以上)
- 経過時間は1秒ごとにクライアントで更新(`start_at` 基準の差分計算。date-fns使用)
- リロード・別デバイス: pageがサーバーで実行中エントリを取得してpropsで渡すことで復元する(ポーリングはしない。再同期・「更新」・操作後の revalidate で最新化)

### 実績ブロックの最小表示(右レーン)

- 日列の**右45%レーン**(P2-1で空けた領域)に、表示範囲内の実績エントリを**濃い塗り**で表示する
- 実行中エントリは `start_at`〜現在時刻のブロックとして表示(現在時刻ラインと同様、ハイドレーション後に描画)
- ブロック内はタイトルのみ(P2-1の予定ブロックと同じクリップ・重複分割・最小高さ規則を再利用)
- 予定とのズレの視覚化・色の作り込み(予定と同系色など)は**P3-1で行う**。ここでは「記録が見える」最小限まで

### 楽観的更新とエラー時の挙動(メッセージはすべて日本語)

- タップ直後にUIを実行中/停止状態へ即時切替 → Server Action失敗時は元の状態に戻してエラー表示
- 連打対策: Server Action実行中は該当操作のボタンを無効化する

| ケース | 挙動 |
|---|---|
| 開始失敗(DBエラー、リトライ後の競合) | 「タイマーを開始できませんでした」表示、UIを元に戻す |
| 停止失敗(DBエラー) | 「タイマーを停止できませんでした」表示、実行中表示を維持 |
| 未認証(セッション切れ) | `/login` へリダイレクト |
| 別デバイスで停止済みの状態で「停止」 | 成功扱い(冪等)。表示を最新化 |

### アクセシビリティ / UI品質(ui-quality Skill準拠)

- 予定ブロックは `button` 化し、`aria-label` に「〜のタイマーを開始/停止」を含める
- 実行中状態は `aria-pressed` で表現。実行中の判別は色+枠線+「記録中」テキスト
- タイマーバーは 375px 幅で横スクロールなし・本体の操作を隠さない高さにする
- コントラスト比 4.5:1 以上、dark: 対応(既存規約どおり)

## スコープ外

- フリータイマー(タイトル入力で開始)= P2-3
- 実績の手動編集・削除 = P2-4
- オーバーレイ表示の作り込み(予定と実績のズレの視覚化、同系色ルール)= P3-1
- 実行中タイマーのリアルタイム同期(ポーリング/Realtime。復元はロード時のみ)
- 通知・バックグラウンド計測(PWA関連はP3-3)

## テストシナリオ

### 単体(lib/timer)

- S1 [単体] 正常系: Given 実行中エントリの `start_at` When 経過時間を整形 Then `0:00:00` / `0:59:59` / `1:23:45` / `25:00:00`(24時間超)の形式で返る(境界: 0秒・1時間・24時間)
- S2 [単体] 正常系: Given 確定済み実績と実行中エントリ(end=現在時刻扱い) When 日ビューの配置計算 Then 右レーン用のブロック位置(P2-1と同じクリップ・最小高さ規則)が返る

### コンポーネント(calendar-view / running-timer-bar)

- S3 [単体] 正常系: Given 実行中なし When 予定ブロックをタップ Then startTimerが予定のID・タイトルで呼ばれ、即時に実行中表示(楽観的更新)になる
- S4 [単体] 正常系: Given その予定のタイマーが実行中 When 同じ予定ブロックをタップ Then stopTimerが呼ばれ、実行中表示が解除される
- S5 [単体] 正常系: Given 実行中エントリのprops When 描画 Then タイマーバーにタイトルと経過時間が表示され、1秒ごとに更新される。停止ボタンでstopTimerが呼ばれる
- S6 [単体] 異常系: Given startTimerが失敗を返す When 予定ブロックをタップ Then 日本語エラーメッセージが表示され、UIが元の状態に戻る
- S7 [単体] 正常系: Given 確定済み実績エントリと実行中エントリ When 日ビューを描画 Then 右レーンに濃い塗りの実績ブロックが表示される(実行中は start〜現在)
- S8 [単体] 正常系: Given 実行中エントリなしのprops When 描画 Then タイマーバーは表示されない

### 結合(lib/timer/service 〜 ローカルDB。RLS認証済みクライアントで実行)

- S9 [結合] 正常系: Given 実行中なし When startTimer Then `time_entries` に行が作られ、`google_event_id`・`title`・`start_at` が設定され `end_at IS NULL`
- S10 [結合] 正常系: Given 実行中エントリあり When 別の予定で startTimer Then 既存行の `end_at` が設定され(自動停止)、新しい行が実行中になる。実行中は常に1本
- S11 [結合] 正常系: Given 実行中エントリあり When stopTimer Then `end_at` が設定され実績として確定する(`end_at >= start_at`)
- S12 [結合] 境界値: Given 実行中エントリなし When stopTimer Then エラーにならず成功する(冪等)
- S13 [結合] 境界値(分離): Given ユーザーBの実行中エントリがある When ユーザーAで startTimer / 実行中取得 Then Bの行は停止されず、見えもしない(RLS)
- S14 [結合] 正常系: Given 表示範囲内・範囲外の実績エントリ When 実績読み取りヘルパーで取得 Then 表示週±1週間の行だけが返る(実行中エントリは期間外でも返す)
