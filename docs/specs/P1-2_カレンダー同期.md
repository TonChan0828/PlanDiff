# 仕様書: P1-2 カレンダー読み取り同期

- ステータス: 承認済み(2026-07-06)
- 関連: docs/要件定義書.md の FR-02 / §7.1 / §7.3、docs/開発計画.md P1-2
- 依存: P1-1(Google認証・google_tokens保存・再認可導線)、P0-6(synced_eventsテーブル)
- 指示資料: なし(docs/指示資料/ を確認済み、2026-07-06時点でREADMEのみ)

## 目的

Googleカレンダー(プライマリのみ)の予定をサーバーサイドで取得し、`synced_events` にキャッシュする。
アクセストークンはrefresh tokenから自動更新し、refresh token失効時はP1-1の再認可導線へ誘導する。
UIは「キャッシュ表示 → バックグラウンド同期 → 反映」の順で描画する(§7.3)。

## 仕様

### 全体構成

```
lib/google/token.ts      … refresh token → access token 更新(Google OAuth token endpoint)
lib/google/calendar.ts   … Calendar API 呼び出し(events.list)+ レスポンス正規化
lib/google/sync-range.ts … 「表示週 ± 1週間」の期間計算ヘルパー(週=月曜始まり、date-fns使用)
app/api/calendar/sync/route.ts … 同期Route Handler(Google API呼び出しはここに集約)
components/calendar-sync.tsx    … クライアント側の同期トリガ(マウント時+手動リフレッシュボタン)
app/(app)/calendar/page.tsx     … プレースホルダーを「キャッシュ済み予定の簡易リスト+同期UI」に更新
```

- Google API・トークンを扱うのはサーバーのみ。クライアントには `synced_events` 由来のデータだけを渡す
- `lib/google/*` は `import "server-only"` を付け、クライアントバンドル混入をビルドエラーで防ぐ(admin.tsと同じパターン)

### トークン自動更新(lib/google/token.ts)

- `POST https://oauth2.googleapis.com/token` に `grant_type=refresh_token` / `refresh_token` / `client_id` / `client_secret` を送り、アクセストークンを取得する
- 環境変数 `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` を使用(`.env.local` 設定済み。**Vercel側への追加が必要** → 補記)
- アクセストークンは**保存しない**(毎同期リクエストで取得。1往復増えるがMVPの同期頻度では十分。キャッシュ化はスコープ外)
- 結果は判別可能な型で返す:
  - 成功: `{ ok: true, accessToken }`
  - `invalid_grant`(refresh token失効・取り消し): `{ ok: false, reason: "reauthorize" }`
  - その他(ネットワーク・5xx等): `{ ok: false, reason: "transient" }`
- トークン値・レスポンスボディをログに出さない(エラー種別のみ)

### Calendar API 呼び出し(lib/google/calendar.ts)

- `GET https://www.googleapis.com/calendar/v3/calendars/primary/events`
- クエリ: `timeMin` / `timeMax` / `singleEvents=true`(繰り返し予定をインスタンス展開)/ `maxResults=2500` / `orderBy=startTime`
- `nextPageToken` があれば追加ページをすべて取得して結合する
- 正規化: `{ googleEventId: item.id, title: item.summary ?? "", startAt, endAt }`(UTCのISO文字列に変換)
- **終日イベント(`start.date` 形式)はMVPでは同期対象外**として読み飛ばす(→ 承認前の確認事項1)
- Google が 401 を返した場合は `reason: "reauthorize"`、その他の失敗は `reason: "transient"` として呼び出し元に返す

### 同期 Route Handler(POST /api/calendar/sync)

リクエスト: `{ "timeMin": "<ISO 8601 UTC>", "timeMax": "<ISO 8601 UTC>" }`
(期間はクライアントが「表示中の週の月曜 00:00 − 1週間 〜 +2週間」をユーザーのタイムゾーンで計算して送る。
サーバーはタイムゾーンを持たないため、期間の妥当性だけを検証する)

処理手順:

1. セッション検証(`lib/supabase/server` の `getUser()`)。未ログイン → **401** `{ error: "unauthorized" }`
2. 期間バリデーション。不正 → **400** `{ error: "invalid_range" }`
   - ISO 8601 として解釈可能であること / `timeMin < timeMax` / 期間が **35日以内**(正規利用は21日。上限は乱用防止)
3. `google_tokens` から refresh token を取得(service role。P1-1の `lib/supabase/admin.ts` に読み取り関数を追加)
   - 行がない → **401** `{ error: "reauthorize" }`(Google APIは呼ばない)
4. アクセストークン更新(上記 token.ts)
   - `reauthorize` → `google_tokens` の行を**削除**し(→ 承認前の確認事項2)、**401** `{ error: "reauthorize" }`
   - `transient` → **502** `{ error: "sync_failed" }`(キャッシュは変更しない)
5. Calendar API で期間内の予定を取得(上記 calendar.ts)
   - `reauthorize` / `transient` は手順4と同じ扱い
6. キャッシュ更新(**ユーザーのRLSクライアント**で書き込み。service roleは使わない):
   - 取得した予定を `synced_events` へ upsert(`onConflict: "user_id,google_event_id"`。`synced_at = now`)
   - **期間内の消えた予定を削除**: 該当ユーザーの `start_at < timeMax AND end_at > timeMin` の行のうち、今回取得した `google_event_id` に含まれないものを削除(期間外の行は触らない)
7. **200** で同期後の期間内キャッシュを返す: `{ "events": [{ id, googleEventId, title, startAt, endAt }] }`(`start_at` 昇順)

### 画面(/calendar プレースホルダーの更新)

P2-1(カレンダービュー)の先回りはしない。同期の疎通確認に必要な最小限のみ:

- Server Component で「今週 ± 1週間」の `synced_events` を読み、日付ごとの簡易リスト表示(タイトル+開始〜終了時刻。ユーザーのタイムゾーンで表示)
- クライアント側(`calendar-sync.tsx`):
  - マウント時に `POST /api/calendar/sync` を1回実行(バックグラウンド同期)し、成功したら `router.refresh()` で再描画
  - 「更新」ボタンで手動リフレッシュ(同期中はボタンを無効化+「同期中…」表示)
  - `401 reauthorize` → `/auth/reauthorize` へリダイレクト(P1-1の導線に接続)
  - その他の失敗 → 「同期に失敗しました。時間をおいてもう一度お試しください」を表示(キャッシュ表示は維持)
- 表示は375px幅で崩れないこと(ui-quality Skill準拠。ただし本格UIはP2-1)

### データ変更

- `synced_events` への upsert / delete(ユーザー自身のRLSクライアント経由)
- `google_tokens` は読み取り+失効時の削除のみ(service role経由)
- スキーマ変更なし(P0-6のテーブル・インデックスをそのまま使用)

### エラー時の挙動(メッセージはすべて日本語)

| ケース | サーバー応答 | クライアント挙動 |
|---|---|---|
| 未ログイン | 401 `unauthorized` | (appレイアウトのガードで通常到達しない)`/login` へ |
| refresh token なし/失効 | 401 `reauthorize` | `/auth/reauthorize` へリダイレクト |
| 期間が不正 | 400 `invalid_range` | 「同期に失敗しました。時間をおいてもう一度お試しください」 |
| Google API 障害・ネットワーク | 502 `sync_failed` | 同上(キャッシュ表示は維持) |
| キャッシュ書き込み失敗(DB障害) | 500 `sync_failed` | 同上 |

### セキュリティ(CLAUDE.md「やってはいけないこと」の再確認)

- アクセストークン・refresh token・Googleレスポンスをクライアント・ログ・エラーメッセージに含めない
- `GOOGLE_CLIENT_SECRET` はサーバー環境変数のみ(`NEXT_PUBLIC_` を付けない)
- スコープは `calendar.events.readonly` のみ。書き込みAPIは呼ばない
- カレンダーデータは表示・ギャップ分析(synced_eventsキャッシュ)以外に保存・加工しない(Limited Use)

## スコープ外

- カレンダービュー本体(日/週タイムライン表示)= P2-1。本項目は簡易リストまで
- 週ナビゲーション(過去週・翌週への移動UI)= P2-1(APIは任意期間に対応済みにする)
- アクセストークンのキャッシュ・並行同期の排他制御・サーバー側レート制限(MVPでは同期はユーザー操作起点のみで頻度が低い)
- webhook / push channel、複数カレンダー、終日イベントの表示(v2以降)
- 同期の差分取得(syncToken方式)。MVPは期間の洗い替えで十分

## テストシナリオ

結合テストは `tests/integration/`(ローカルSupabase)を使用。Google API(token endpoint / events.list)は
`fetch` レベルでモックし、DBへの効果はローカルSupabaseで実際に検証する(P1-1と同じ方針)。

### 単体(lib/google)

- S1 [単体] 正常系: Given 有効なrefresh token When `refreshAccessToken` を呼ぶ Then `grant_type=refresh_token`・client_id・client_secret を含むPOSTが送られ、access tokenが返る
- S2 [単体] 異常系: Given tokenエンドポイントが `invalid_grant` を返す When 呼ぶ Then `reason: "reauthorize"` が返り、ログにトークン値が含まれない
- S3 [単体] 異常系: Given tokenエンドポイントがネットワークエラー/5xx When 呼ぶ Then `reason: "transient"` が返る
- S4 [単体] 正常系: Given アクセストークンと期間 When `fetchPrimaryEvents` を呼ぶ Then `primary` カレンダーに `singleEvents=true`・`timeMin`・`timeMax` 付きでリクエストされ、`dateTime` がUTC ISOに正規化されて返る
- S5 [単体] 境界値: Given `nextPageToken` 付きレスポンス(2ページ) When 呼ぶ Then 全ページ分のイベントが結合されて返る
- S6 [単体] 境界値: Given 終日イベント(`start.date`)とタイトルなしイベントが混在 When 呼ぶ Then 終日イベントは除外され、タイトルなしは `""` になる
- S7 [単体] 異常系: Given Calendar APIが401を返す When 呼ぶ Then `reason: "reauthorize"` が返る
- S8 [単体] 境界値: Given 期間計算ヘルパーに任意の日付 When 呼ぶ Then その週の月曜00:00(ユーザーTZ)の−7日〜+14日のUTC期間が返る

### 結合(Route Handler 〜 ローカルDB)

- S9 [結合] 正常系: Given ログイン済み+google_tokens行あり、Googleモックが2件返す When `/api/calendar/sync` を呼ぶ Then `synced_events` に2行保存され(`synced_at` 更新)、200で `events` 2件が `start_at` 昇順で返る
- S10 [結合] 正常系(更新): Given 既に同じ `google_event_id` の行がある When タイトル・時刻が変わった内容で同期 Then 行数は増えず、内容と `synced_at` が更新される
- S11 [結合] 正常系(削除): Given 期間内のキャッシュ行(Google側で削除済み)と期間外のキャッシュ行がある When 同期 Then 期間内の消えた行だけ削除され、期間外の行は残る
- S12 [結合] 異常系: Given 未ログイン When 呼ぶ Then 401、DBは変更されない
- S13 [結合] 異常系: Given google_tokens に行がない When 呼ぶ Then 401 `reauthorize`、Google APIは呼ばれない
- S14 [結合] 異常系: Given tokenエンドポイントが `invalid_grant` When 呼ぶ Then 401 `reauthorize`、google_tokens の行が削除される(確認事項2の決定に従う)
- S15 [結合] 異常系: Given Calendar APIが5xx When 呼ぶ Then 502 `sync_failed`、キャッシュは変更されない
- S16 [単体] 異常系: Given `timeMin >= timeMax` / 35日超 / 不正な日付文字列 When 呼ぶ Then 400 `invalid_range`
- S17 [結合] 境界値(分離): Given 別ユーザーのキャッシュ行が同期間に存在 When ユーザーAが同期 Then 別ユーザーの行は削除・変更されない
- S18 [単体] 正常系(コンポーネント): Given キャッシュ済み予定 When /calendar を描画 Then 予定リストと「更新」ボタンが表示され、同期失敗時は日本語エラーメッセージが表示される

### 手動疎通確認(自動テストの補完。DoDに含める)

- M1: 実Googleアカウントでログイン → /calendar で今週±1週間の実予定が表示される
- M2: Googleカレンダー側で予定を追加・変更・削除 → 「更新」ボタンで反映される
- M3: `google_tokens` の行を手動削除した状態で同期 → `/auth/reauthorize` に誘導される

## 確認事項の決定(2026-07-06 ユーザー承認)

1. **終日イベントの扱い**: MVPでは同期対象外(読み飛ばし)で確定。
   理由: 予定vs実績のオーバーレイは時刻ベースであり、終日イベント(休暇・記念日等)は計画時間の比較対象にならないため。v2で再検討
2. **refresh token失効(invalid_grant)時の `google_tokens` 行削除**: 削除するで確定。
   理由: 「行がある=使えるトークン」という状態を単純に保てる(死んだトークンを残さない)。再認可すればP1-1のcallbackで再upsertされる
