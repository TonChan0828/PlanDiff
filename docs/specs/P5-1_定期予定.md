## 目的

毎日の朝会・週次MTG・日次作業のような定期的な予定を、毎回手で登録せずに済むようにする。
アプリ内予定(FR-10)の拡張として「毎日 / 毎週(曜日複数)/ 平日」の繰り返しルールを持ち、
既存のコアループ(予定連動タイマー FR-04・オーバーレイ FR-06・ギャップサマリー FR-07)が
繰り返し予定に対しても無改修で機能することを狙う。

### 決定済みの要件(brainstorming結果)

| 論点 | 決定 |
|---|---|
| 繰り返しパターン | 毎日 / 毎週(曜日複数選択)/ 平日(月〜金)の3種 |
| 編集・削除の粒度 | 「この回のみ」+「繰り返し全体」の2択(「これ以降」は作らない) |
| 期限 | 無期限がデフォルト。任意で終了日を指定可能 |
| 実現方式 | 案A: ルールテーブル+表示範囲のオンデマンド実体化(冪等insert) |
| 全体削除時の過去分 | 過去のインスタンスは残す(計画していた事実を保持。ユーザー承認済み) |

## 仕様

### アーキテクチャ概要(案A: ルール+実体化)

```
recurring_rules(繰り返し定義)
     │  materializeRecurringInstances(表示範囲の読み込み時に実行)
     ▼
synced_events に source='app'・google_event_id='rec:<ruleId>:<YYYY-MM-DD>' で冪等insert
     │
     ▼
既存のタイマー / オーバーレイ / ギャップサマリーがそのまま動く(改修なし)
```

- 実体化のキーは決定的(`rec:<ruleId>:<発生日>`)で、`unique (user_id, google_event_id)` と
  `ON CONFLICT DO NOTHING` により何度実行しても重複しない
- 既存行(=「この回のみ」編集済みの行を含む)は上書きしないため、個別編集が自然に例外になる
- 「この回のみ削除」は `recurring_exceptions`(tombstone)に記録し、再実体化での復活を防ぐ

### データ変更(マイグレーション。実装時は db-migration Skill に従う)

```sql
-- P5-1: 繰り返し予定のルールと例外(この回のみ削除)の2テーブル

create table public.recurring_rules (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users (id) on delete cascade,
  title text not null,
  pattern text not null
    constraint recurring_rules_pattern_check check (pattern in ('daily', 'weekly', 'weekdays')),
  -- weekly のときのみ使用。0=日曜〜6=土曜(JSのDate#getDayと一致させる)
  weekdays smallint[]
    constraint recurring_rules_weekdays_range check (
      weekdays is null or (weekdays <@ array[0,1,2,3,4,5,6]::smallint[])
    ),
  constraint recurring_rules_weekdays_required check (
    pattern <> 'weekly'
    or (weekdays is not null and array_length(weekdays, 1) between 1 and 7)
  ),
  -- 「毎朝9時」はローカル時刻の概念のため、ルールはローカル時刻+IANAタイムゾーンで持ち、
  -- 実体化の瞬間にUTC(timestamptz)へ変換する(CLAUDE.mdのUTC保存原則はsynced_events側で維持)
  start_time time not null,
  end_time time not null,
  constraint recurring_rules_time_check check (start_time < end_time),
  timezone text not null,
  starts_on date not null,
  ends_on date,
  constraint recurring_rules_ends_check check (ends_on is null or ends_on >= starts_on),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index recurring_rules_user_idx on public.recurring_rules (user_id);

create table public.recurring_exceptions (
  id uuid primary key default gen_random_uuid(),
  rule_id uuid not null references public.recurring_rules (id) on delete cascade,
  user_id uuid not null references auth.users (id) on delete cascade,
  occurrence_date date not null,
  created_at timestamptz not null default now(),
  constraint recurring_exceptions_unique unique (rule_id, occurrence_date)
);

create index recurring_exceptions_user_idx on public.recurring_exceptions (user_id);
```

- RLS: 両テーブルとも有効化し、own-row CRUDポリシー(`synced_events` と同一パターン。
  `select/insert/update/delete` それぞれ `auth.uid() = user_id`)。GRANTは
  `authenticated` に select/insert/update/delete、`service_role` に all(既存踏襲)
- `updated_at` トリガー: 既存の `private.set_updated_at()` を `recurring_rules` に適用
- 日をまたぐ繰り返し(22:00〜翌1:00)はスコープ外(`start_time < end_time` 制約)
- 依存追加: `@date-fns/tz`(date-fns v4 公式のTZコンパニオン。TZDateでローカル時刻→UTC変換)

### 展開・実体化ロジック(`lib/calendar/recurring.ts`)

**純粋関数 `expandOccurrences(rule, rangeStartUtc, rangeEndUtc): Occurrence[]`**

- ルールのタイムゾーンで発生日(date)を列挙し、各発生日について
  `startAt/endAt`(UTC ISO)と `occurrenceDate`(`YYYY-MM-DD`)を返す
- 発生条件: `starts_on <= 発生日 <= (ends_on ?? ∞)` かつパターンに一致
  (daily=毎日、weekdays=月〜金、weekly=weekdays配列に発生日の曜日が含まれる)
- 範囲判定は実体化後のUTC時刻で `startAt < rangeEnd && endAt > rangeStart`
  (`fetchSyncedEvents` の重なり判定と同一)

**実体化 `materializeRecurringInstances(client, baseDate): Promise<void>`**

- 対象範囲は `computeSyncRange(baseDate)`(表示週±1週。既存の取得範囲と一致)
- 手順: ①本人の `recurring_rules` を取得 → ②各ルールを `expandOccurrences` で展開 →
  ③本人の `recurring_exceptions` にある `(rule_id, occurrence_date)` を除外 →
  ④ `synced_events` へ `source='app'`・`google_event_id='rec:<ruleId>:<occurrenceDate>'`・
  `title=ルールのtitle` で **upsert(`onConflict: 'user_id,google_event_id'`,
  `ignoreDuplicates: true`)** — 既存行は一切変更しない
- 呼び出し箇所: `/calendar`・`/track`・`/summary` の各pageで `fetchSyncedEvents` の**直前**に実行
  (Server Component内。ルールが0件なら実質no-op)
- 失敗時: 実体化の失敗でページ全体を落とさない(エラーはログのみ、予定表示は既存分で継続)

### ルールのCRUD(ロジック層+Server Action)

- ロジック: `lib/calendar/recurring.ts`(`app-events.ts` のパターン踏襲。
  `{ ok: boolean }` を返す)
- Server Action: `app/(app)/calendar/event-actions.ts` に追加
  (認証済みRLSクライアント。service role不使用)

| 関数 | 入力 | 挙動 |
|---|---|---|
| `createRecurringRule` | title, pattern, weekdays?, startTime, endTime, timezone, startsOn, endsOn? | バリデーション後insert。その場では実体化しない(保存成功→`router.refresh()`→pageの実体化で表示される) |
| `updateRecurringRule` | ruleId, 同上のフィールド | ルールをupdate → **今日以降**の実体化済み行(`google_event_id like 'rec:<ruleId>:%'` かつ `start_at >= 今日のルールTZでの0時(UTC変換)`)を削除 → 今日以降の `recurring_exceptions` も削除(再生成でルール通りに戻る)。過去の行は触らない |
| `deleteRecurringRule` | ruleId | **今日以降**のインスタンス行を削除 → ルールをdelete(cascadeでexceptionsも消える)。**過去のインスタンスは残す**(過去の計画vs実績を保持) |
| `deleteRecurringOccurrence` | eventId(synced_eventsのid) | 対象行の `google_event_id` から `rec:<ruleId>:<date>` を解析 → `recurring_exceptions` へinsert(unique違反=登録済みは成功扱い)→ 行をdelete |

- サーバー側バリデーション(不合格は `{ ok: false }`、書き込まない):
  - title: trim後1〜200文字(P2-5と同一)
  - pattern: 3種のいずれか。weekly時はweekdaysが1〜7個・値0〜6・重複なし
  - startTime < endTime(`HH:mm` 形式)
  - timezone: `Intl.supportedValuesOf('timeZone')` 相当の妥当なIANA名
    (実装は `new Intl.DateTimeFormat` で解決可能かで判定)
  - startsOn: 妥当な日付。endsOn指定時は endsOn >= startsOn
- 「この回のみ」の**編集**は既存の `updateAppEvent` をそのまま使う(`rec:` 行も
  `source='app'` のため。google_event_id は変更されず、再実体化で上書きもされない)
- 全体編集で「この回のみ」の変更(編集・削除)が**今日以降分は失われる**のは仕様
  (UIで警告文言を表示)
- タイマーとの関係: `time_entries.google_event_id` に `rec:` キーがスナップショットされる。
  ルール削除・この回のみ削除の後も実績は残り、対応する予定行が無ければ既存仕様どおり
  「割り込み実績」として集計される(P2-5と同一の挙動)

### UI(実装時は ui-quality Skill・375px確認)

**`components/app-event-panel.tsx` の拡張(作成モード)**

- 「繰り返し」セレクトを追加: `なし(既定)/ 毎日 / 毎週 / 平日`
- 「毎週」選択時のみ曜日チップ(日〜土、複数選択可)を表示。初期値は開始日時の曜日
- 繰り返し≠なしのとき「終了日(任意)」のdate入力を表示。空=無期限
- 繰り返し≠なしのとき、既存の開始/終了datetime-localは「初回の開始・終了」と解釈する:
  開始日時の日付→`startsOn`、開始・終了の時刻→`startTime`/`endTime`。
  開始と終了が同日でない場合はフォーム内エラー(日またぎ繰り返しはスコープ外)
- タイムゾーンは端末の `Intl.DateTimeFormat().resolvedOptions().timeZone` を送る
- クライアント側バリデーション(Server Actionを呼ばない): 既存の検証に加え、
  毎週で曜日未選択 / 終了日 < 開始日 / 繰り返し時の日またぎ

**`rec:` 予定の編集・削除(2択ステップ)**

- `rec:` 予定の編集ボタンをタップすると、パネル冒頭に選択ステップを表示:
  「この予定のみ」/「繰り返し全体」
- **この予定のみ** → 既存の編集モード(タイトル・開始・終了)。削除ボタンは
  `deleteRecurringOccurrenceAction` を呼ぶ(確認ステップはP2-4/P2-5と同一UX)
- **繰り返し全体** → ルール編集モード(タイトル・時刻・パターン・曜日・終了日。
  開始日は変更不可)。「今日以降の個別の変更は失われます」の注意文を表示。
  削除ボタンは `deleteRecurringRuleAction`(確認ステップ+「過去の予定は残ります」注記)
- ルール編集モードの初期値はサーバーから取得したルール
  (`/calendar` pageで本人のルール一覧をpropsとして渡す)
- 単発のアプリ予定(`app:`)・Google予定の既存挙動は変更しない

**カレンダー表示**

- `rec:` 予定のブロックに繰り返しを示す小さなマーク(「⟳」等)をタイトル横に表示
  (それ以外の見た目・タップ挙動=タイマー開始/停止は既存のまま)

### エラー時の挙動一覧(すべて日本語。`lib/calendar/messages.ts` に集約)

| ケース | 表示 |
|---|---|
| 毎週で曜日未選択 | フォーム内「繰り返す曜日を選択してください」 |
| 繰り返し予定で開始と終了が別日 | フォーム内「繰り返し予定は同じ日の時間帯で入力してください」 |
| 終了日 < 開始日 | フォーム内「繰り返しの終了日は開始日以降にしてください」 |
| ルール作成 Server Action 失敗 | パネル内「繰り返し予定の作成に失敗しました」 |
| ルール更新 Server Action 失敗 | パネル内「繰り返し予定の更新に失敗しました」 |
| ルール削除 Server Action 失敗 | パネル内「繰り返し予定の削除に失敗しました」 |
| この回のみ削除 失敗 | パネル内「予定の削除に失敗しました」(P2-5文言を流用) |
| タイトル空・日時未入力・終了≤開始 | P2-5の既存文言を流用 |

### セキュリティ

- ルール・例外の読み書きはユーザー本人のRLSクライアントのみ(service role不使用)
- `deleteRecurringOccurrence` / インスタンス削除は `.eq("source", "app")` +
  `google_event_id like 'rec:%'` 条件でGoogleキャッシュ行に触れない
- timezone文字列はバリデーションして保存(不正値での実体化例外を防ぐ)

## スコープ外

- 「これ以降すべて」の編集・削除(シリーズ分割)
- 毎月・隔週・N日ごとのパターン
- 日をまたぐ繰り返し予定(単発予定では従来どおり可能)
- 繰り返しの終了日をUIから外す/付け直す際の過去分再計算(ends_on変更は今日以降にのみ反映)
- Googleカレンダーの繰り返し予定(RRULE)の解釈変更(同期は従来どおり展開済み
  インスタンスを受け取るのみ)
- 実体化行の自動クリーンアップ(表示済み週の行は残る。容量影響は軽微)
