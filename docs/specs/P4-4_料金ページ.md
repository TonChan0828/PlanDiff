## 目的

- MVPでは課金を実装せず、料金ページで「Pro近日公開」を提示して
  **課金意欲(クリック率)の基準値を収集**する(要件定義書 §10・§13)
- ベータ投入(P4-6)前に、LPから料金情報へ辿れる導線を成立させる

## 仕様

### P4-4a. 計測テーブル(DBマイグレーション)

`pro_interest_events` テーブルを新設する。**個人情報は保存しない**
(user_id・IP・UA等の列を持たない。プライバシーポリシー改定を不要にするため)。

```sql
create table public.pro_interest_events (
  id uuid primary key default gen_random_uuid(),
  event_type text not null check (event_type in ('view', 'click')),
  created_at timestamptz not null default now()
);
```

- `view` = 料金ページの閲覧、`click` = 「Proに興味あり」ボタンのクリック。
  クリック率 = click数 / view数 で算出できる(集計UIは作らない。Supabaseで直接クエリ)
- **RLS**: 有効化し、**ポリシーは一切付けない**(anon/authenticatedから読み書き不可)。
  書き込みはservice role経由のRoute Handlerのみ(`google_tokens` と同じパターン)
- P4-2のデータ全削除との関係: 本テーブルは個人に紐づかない統計のため削除対象外

### P4-4b. 記録API(Route Handler)

`POST /api/pro-interest`(認証不要。未ログイン訪問者も計測対象)

- リクエスト: `{ "event": "view" | "click" }`(JSON)
- 挙動: バリデーション後、service roleクライアントで1行INSERTし `204 No Content`
- エラー: 不正なbody・不明なevent → `400`(`{ "error": "不正なリクエストです" }`)。
  DB障害 → `500`(`{ "error": "記録に失敗しました" }`)。トークン・内部情報は返さない
- レート制限・bot除外はMVPではスコープ外(基準値収集が目的のため許容。仕様書に明記)

### P4-4c. 料金ページ(`app/(marketing)/pricing/page.tsx`)

marketingレイアウト配下に新設。モバイルファースト(375px)・ダーク対応(D-1トークン準拠)。

構成(上から):

1. **見出し**: 「料金」+サブコピー(ベータ期間中はすべての機能を無料で利用可、の趣旨)
2. **プラン2枚**(モバイルは縦積み、sm以上で横並び):
   - **Free(現在)**: 「¥0」。タイマー・記録 / 予定×実績のオーバーレイ /
     日・週のギャップサマリー / アプリ内の予定管理 — **現在提供中の機能のみ**を列挙する
     (履歴2週間制限など未実装の制限は書かない)
   - **Pro(近日公開)**: 価格は表示しない(「価格は近日公開」)。
     構想中の機能を「予定」と明記して列挙: 履歴無制限 / タスク種別ごとの
     見積もり傾向分析・サジェスト / 集中時間・会議コスト分析 / 複数カレンダー /
     エクスポート・API(要件定義書 §10)。
     ※開発ツール連携(GitHub等)は時期未定のため載せない。
     Googleカレンダー連携は凍結中のため「提供中」と書かない
3. **「Proに興味あり」ボタン**(Proカード内・主要CTA):
   - クリックで `POST /api/pro-interest`(`click`)を送信。送信中はdisabled(連打防止)
   - 成功: ボタンを「興味を受け付けました。ありがとうございます!」の完了表示
     (disabled)に置き換え、`localStorage` にフラグを保存。次回訪問時も完了表示
   - 失敗: 日本語のエラーメッセージを表示し、ボタンは再度押せる状態に戻す
4. **Freeで始めるCTA**: 「無料で始める」→ `/signup`

- ページ表示時に `POST /api/pro-interest`(`view`)をクライアントから1回送信する
  (静的化されるためサーバー側では記録しない)。view送信の失敗はUIに出さず無視する
  (計測欠損として許容。ユーザー体験を妨げない)
- `metadata` の title / description を設定する(例: 「料金 | PlanDiff」)

### P4-4d. 導線

- marketingフッター(`app/(marketing)/layout.tsx` の `FOOTER_LINKS`)に
  「料金」→ `/pricing` を追加(プライバシーポリシーの前)
- LP(`app/(marketing)/page.tsx`)のベータ案内セクションに
  「料金(Pro近日公開)を見る」テキストリンクを追加

## スコープ外

- 課金実装・決済連携(v2以降。要件定義書 §4.2)
- Pro価格の決定・表示(§14の未決事項のまま)
- ウェイトリスト(メールアドレス収集)・個人単位の計測(user_id保存)
- 集計ダッシュボードUI(Supabaseで直接クエリする)
- レート制限・bot除外(基準値収集が目的のため許容)
- LPの料金リンク追加以外の変更

## テストシナリオ

- S1 [単体]: Given 料金ページ When 描画 Then 見出し「料金」、Free(¥0)と
  Pro(近日公開)の2プラン、「Proに興味あり」ボタン、「無料で始める」(→/signup)が
  表示され、Proの機能に「予定」の明記がある
- S2 [単体]: Given 未クリック状態 When 「Proに興味あり」をクリック
  Then `POST /api/pro-interest`(click)が1回呼ばれ、成功後にお礼の完了表示
  (disabled)へ変わり、localStorageにフラグが保存される
- S3 [単体]: Given クリック送信中(レスポンス未返却) When 再度クリック
  Then 追加のPOSTは発生しない(連打防止・境界値)
- S4 [単体]: Given POSTが500で失敗 When クリック Then 日本語のエラーメッセージが
  表示され、ボタンが再度押せる状態に戻る(異常系)
- S5 [単体]: Given localStorageに興味ありフラグ When 描画
  Then 最初から完了表示になり、clickのPOSTは発生しない
- S6 [単体]: Given 料金ページ When マウント Then viewイベントのPOSTが1回送信される。
  失敗してもエラーUIは表示されない
- S7 [単体]: Given marketingフッター When 描画 Then 「料金」リンク(→/pricing)が
  表示される(既存のプライバシーポリシー・利用規約リンクも維持)
- S8 [結合]: Given `POST /api/pro-interest` に `{event:'click'}` When 実行
  Then `pro_interest_events` に `event_type='click'` の行が1件増え、204が返る
  (viewも同様)
- S9 [結合]: Given 不正なbody(`{}`・`{event:'other'}`・非JSON) When POST
  Then 400と日本語エラーが返り、行は増えない(異常系・境界値)
- S10 [結合]: Given anonキーのSupabaseクライアント When `pro_interest_events` を
  SELECT / INSERT Then RLSにより拒否される(0件・エラー)
