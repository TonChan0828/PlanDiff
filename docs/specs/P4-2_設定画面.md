# 仕様書: P4-2 設定画面の完成(アカウント表示・ログアウト・データ全削除)

- ステータス: 承認済み(2026-07-11)
- 関連: docs/要件定義書.md の §7 画面一覧「設定」・§9 データ削除要件(Limited Use)/ docs/specs/P1-3_メール認証とGoogle任意連携.md(/settings の骨格)/ docs/specs/P2-5_アプリ内予定とGoogle連携凍結.md(Google連携フラグ)
- 指示資料: docs/指示資料/ に本タスク関連の資料なし(2026-07-11 確認)

## 目的

設定画面を「アカウント確認・ログアウト・データ全削除」ができる状態に完成させる。
データ全削除は Google OAuth 審査要件(Limited Use)であり、凍結解除・審査申請(P0-5)の前提にもなる。

現状の `/settings` は Google 連携セクションのみで、凍結中(`GOOGLE_INTEGRATION_ENABLED` 未設定)は
実質空ページになっている。

## 仕様

### 1. アカウントセクション(新規)

- ログイン中ユーザーのメールアドレスを表示する
- 見出し「アカウント」

### 2. ログアウト(新規)

- 設定画面にログアウトボタンを置く。既存の `signOutAction`(`app/(app)/actions.ts`)を再利用する
- 実行後は `/login` へリダイレクト(既存挙動のまま)

### 3. データ全削除(新規・本タスクの中心)

- 「危険な操作」セクションを画面最下部に置き、説明文で以下を明示する:
  - アカウントと、予定・実績などすべてのデータが完全に削除され、復元できないこと
- **誤操作防止**: 確認用テキスト入力に「削除」と正確に入力するまで削除ボタンを無効化する
  (クライアント葉コンポーネント `components/delete-account-section.tsx` として実装)
- **削除処理**(Server Action `deleteAccountAction`):
  1. `supabase.auth.getUser()` で本人確認。未認証なら `/login` へリダイレクト
  2. service role クライアントで `auth.admin.deleteUser(userId)` を実行
     - スキーマは全テーブル(`profiles` / `google_tokens` / `synced_events` / `time_entries`)が
       `auth.users` への `on delete cascade` のため、この1操作で全データが削除される
     - 実行中タイマー(`end_at IS NULL`)やアプリ内予定(`source='app'`)が残っていても削除できる
  3. `supabase.auth.signOut()` でセッション破棄
  4. `/login?deleted=1` へリダイレクト
- **削除失敗時**: `/settings?error=account_delete_failed` へリダイレクトし、
  「アカウントの削除に失敗しました。時間をおいてもう一度お試しください」を表示する
  (エラー詳細・トークン類はログ・画面に出さない)
- **ログイン画面**: `?deleted=1` のとき「アカウントとデータをすべて削除しました」を表示する

### 4. Google連携セクション(変更なし)

- P1-3/P2-5 の実装・凍結挙動をそのまま維持する(凍結中は非表示、フラグONで連携状態・解除を表示)
- 再認可導線は既存の「連携する」ボタン(`/api/google/connect`)のままとし、本タスクでは変更しない

### UI文言

- 文言はすべて `lib/settings/messages.ts`(および login 用は既存の messages 集約先)に追加する(日本語)
- モバイルファースト。375px 幅で入力・ボタンが崩れないこと(`ui-quality` チェックリスト)

### データ変更

- スキーマ変更なし(マイグレーション不要)。既存の cascade 設計(P0-6)を利用する

## スコープ外

- パスワード変更・メールアドレス変更(要件定義書の設定画面スコープに含まれない。必要になれば別タスク)
- プロフィール編集(表示名など)
- Google 再認可フローの変更・凍結解除
- 削除前のデータエクスポート

## テストシナリオ

<!-- 対応テスト: tests/app/(app)/settings/, tests/components/, tests/integration/ -->

正常系:

- S1 [単体]: Given ログイン済みユーザー When `/settings` を表示 Then アカウントセクションにメールアドレスが表示される
- S2 [単体]: Given `/settings` 表示中 When ログアウトボタンを見る Then `signOutAction` を action に持つフォームが存在する
- S3 [単体]: Given 削除セクション When 確認欄に「削除」と入力 Then 削除ボタンが有効になる
- S4 [結合]: Given データ一式(profiles・synced_events(source='app'含む)・time_entries・google_tokens)を持つユーザー When アカウント削除を実行 Then `auth.users` と全テーブルの当該ユーザー行が削除される
- S5 [結合]: Given 2ユーザーが存在 When 一方がアカウント削除 Then 他方のデータは影響を受けない
- S6 [単体]: Given `/login?deleted=1` Then 「アカウントとデータをすべて削除しました」が表示される

異常系:

- S7 [単体]: Given 未認証状態 When `deleteAccountAction` を実行 Then `/login` へリダイレクトされ削除処理は呼ばれない
- S8 [単体]: Given `auth.admin.deleteUser` が失敗 When 削除を実行 Then `/settings?error=account_delete_failed` へリダイレクトされる
- S9 [単体]: Given `/settings?error=account_delete_failed` Then 日本語のエラーメッセージが表示される

境界値:

- S10 [単体]: Given 確認欄が空・「削」・「削除 」(余分な空白) Then 削除ボタンは無効のまま
- S11 [結合]: Given 実行中タイマー(`end_at IS NULL`)を持つユーザー When アカウント削除 Then エラーなく削除される
- S12 [単体]: Given 凍結中(`GOOGLE_INTEGRATION_ENABLED` 未設定)の `/settings` Then Google連携セクションは表示されず、アカウント・ログアウト・削除セクションは表示される(P2-5凍結挙動の回帰確認)
