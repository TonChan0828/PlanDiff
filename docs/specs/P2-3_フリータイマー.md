# 仕様書: P2-3 フリータイマー

- ステータス: 承認済み(2026-07-07)
- 関連: docs/要件定義書.md の FR-05 / 開発計画.md P2-3
- 依存: P2-2(予定連動タイマー。`lib/timer/service.ts` の `startTimer` は既に `googleEventId: null` を受け付ける設計)
- 指示資料: なし(docs/指示資料/ を確認済み、2026-07-07時点でREADMEのみ)

## 目的

カレンダーに予定がない作業(割り込み対応、雑務など)も記録できるように、タイトルを自由入力してタイマーを開始/停止できるようにする。
既存の予定連動タイマー(P2-2)と同じ `time_entries` テーブル・同じ実行中1本制約をそのまま使う。`google_event_id` が `null` の行として保存する点のみが違い。

## 仕様

### 全体構成(変更/追加ファイル)

```
components/free-timer-bar.tsx        … 新規。タイトル入力+開始ボタン(クライアント)
components/calendar-view.tsx         … 変更。実行中なし時に FreeTimerBar を表示、開始ハンドラを追加
lib/timer/messages.ts                … 変更。フリータイマー用の文言を追加
lib/timer/service.ts                 … 変更なし(既に googleEventId: null を受け付ける)
app/(app)/calendar/timer-actions.ts  … 変更なし(startTimerAction はそのまま使える)
```

- 新規のServer Action・スキーマ変更・マイグレーションは**なし**。既存の `startTimer` / `startTimerAction` をタイトル入力から呼ぶだけ
- Google APIは使わない(P2-2と同様)

### 表示位置と切り替え

- カレンダー画面下部の固定(sticky)バー領域は、実行中タイマーの有無で以下のどちらかを表示する(同じ場所に排他表示):
  - 実行中あり → 既存の `RunningTimerBar`(タイトル+経過時間+停止ボタン)
  - 実行中なし → 新規の `FreeTimerBar`(タイトル入力+開始ボタン)
- 実行中タイマーが予定連動(`googleEventId` あり)かフリー(`null`)かに関わらず、実行中は常に `RunningTimerBar` を表示する(表示コンポーネントはP2-2から変更しない)
- 実行中に別のフリータイマーを直接始める導線は設けない(停止 → `FreeTimerBar` で新規開始、の順になる)。これはP2-2の「予定連動タイマーは実行中は常にRunningTimerBar」という既存方針と一貫させるための意図的なスコープ限定

### タイトル入力

- 入力は任意(空欄可)。空欄のまま開始した場合は `title: ""` としてそのまま保存する(予定連動タイマーで予定タイトルが空の場合と同じ扱い。表示は既存の「(タイトルなし)」に委ねる)
- 送信直前に前後の空白を `trim()` する(全角/半角スペースのみの入力は空欄扱いになる)
- 文字数上限は設けない(既存の `time_entries.title` にDB制約がないため。予定タイトルも無制限)
- 開始操作: 「開始」ボタンのクリック、またはinput内でのEnter(`<form onSubmit>`)のどちらでも同じ挙動

### 開始/停止の挙動

| 操作 | 挙動 |
|---|---|
| 入力欄にタイトルを入力(または空欄のまま)して開始 | `startTimerAction({ googleEventId: null, title })` を呼ぶ。P2-2と同じ楽観的更新(即時に`RunningTimerBar`表示へ切替 → 成功で`router.refresh()`、失敗でロールバック) |
| 実行中のフリータイマーを停止 | 既存の`RunningTimerBar`の停止ボタン → `stopTimerAction()`(P2-2と同じ、変更なし) |
| 別デバイスで予定連動タイマーが実行中の状態でこの画面からフリータイマーを開始 | サーバー側(`startTimer`)が既存を自動停止してから新規行を作る(P2-2の自動停止ロジックがそのまま適用される。予定→フリー、フリー→予定のどちらの向きも対応済みのロジックだが、これまでのテストは予定→予定のみだったため本仕様で明示的にテストする) |

### エラー時の挙動

- 開始失敗時: 既存の `T.startError`(「タイマーを開始できませんでした」)を表示し、`FreeTimerBar` に戻る(入力したタイトルは保持しない。再入力してもらう)
- P2-2と同じ場所(`timerError` 表示領域)を共用する

### アクセシビリティ / UI品質(ui-quality Skill準拠)

- 入力欄: `aria-label="作業内容(空欄可)"`、`placeholder="作業内容(空欄可)"`
- 開始ボタン: 最小タップターゲット44px、`aria-label="フリータイマーを開始"`
- `FreeTimerBar` は `RunningTimerBar` と同じ位置・同じ角丸/枠線スタイル(`data-testid="free-timer-bar"`)にして、実行中/未実行の切り替えで画面のガタつきを抑える
- 375px幅で横スクロールなし。入力欄は `flex-1` で伸縮し、ボタンは折り返さない
- dark: 対応、コントラスト比 4.5:1 以上(既存規約どおり)

## スコープ外

- 実行中の別タイマーから直接フリータイマーへ切り替えるUI導線(停止してから開始する運線のみ提供)
- タイトルのオートコンプリート・過去タイトルからの候補表示
- 実績の手動編集・削除(P2-4)
- オーバーレイ表示の作り込み(P3-1)

## テストシナリオ

### 単体(components/free-timer-bar)

- S1 [単体] 正常系: Given 実行中なし When タイトルを入力し開始ボタンをタップ Then `onStart` がトリムされたタイトルで呼ばれる
- S2 [単体] 境界値: Given タイトル未入力(空欄) When 開始ボタンをタップ Then `onStart` が `""` で呼ばれる(空タイトルでも開始できる)
- S3 [単体] 境界値: Given タイトル前後に空白を含む入力(`"  読書  "`) When 開始 Then トリム済みの `"読書"` で `onStart` が呼ばれる
- S4 [単体] 正常系: Given `pending=true` When 描画 Then 入力欄と開始ボタンが無効化される(連打対策)

### 単体(calendar-view統合)

- S5 [単体] 正常系: Given 実行中なし When `FreeTimerBar` にタイトルを入力し開始 Then `startTimerAction` が `{ googleEventId: null, title }` で呼ばれ、即時に `RunningTimerBar`(実行中表示)へ切り替わる(楽観的更新)
- S6 [単体] 正常系: Given 実行中なし When タイトル未入力で開始 Then `startTimerAction` が `{ googleEventId: null, title: "" }` で呼ばれ、実行中表示は「(タイトルなし)」になる
- S7 [単体] 異常系: Given `startTimerAction` が失敗を返す When フリータイマーを開始 Then 日本語エラーメッセージが表示され、`FreeTimerBar` の表示に戻る(実行中表示にならない)
- S8 [単体] 正常系: Given 実行中エントリ(`googleEventId: null` のフリータイマー) When 描画 Then `RunningTimerBar` が表示され、`FreeTimerBar` は表示されない。停止すると `FreeTimerBar` に戻る

### 結合(lib/timer/service 〜 ローカルDB。RLS認証済みクライアントで実行)

- S9 [結合] 正常系: Given 実行中なし When `startTimer(client, { googleEventId: null, title: "読書" })` Then `time_entries` に `google_event_id IS NULL` の行が作られ実行中になる
- S10 [結合] 正常系: Given 予定連動タイマーが実行中 When フリータイマーで `startTimer` Then 既存の予定連動タイマーが自動停止され(`end_at` 設定)、フリータイマーが実行中になる(`google_event_id IS NULL`)。実行中は常に1本
- S11 [結合] 正常系: Given フリータイマーが実行中 When 別の予定で `startTimer` Then フリータイマーが自動停止され実績として確定する(`google_event_id IS NULL` のまま `end_at` が設定される)

### 手動確認(DoDに含める)

- M1: 実環境でフリータイマー入力にタイトルを入れて開始 → リロードで復元 → 停止 → 実績が右レーンに表示される、の一連を確認(375px)
- M2: タイトル未入力のまま開始 → 停止 → 実績ブロックに「(タイトルなし)」が表示されることを確認
