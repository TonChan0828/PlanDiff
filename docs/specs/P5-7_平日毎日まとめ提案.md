# 仕様書: P5-7 実績からの「平日/毎日/複数曜日」まとめ提案

- ステータス: 承認済み(2026-07-23)
- 関連: docs/要件定義書.md の FR-13 / docs/開発計画.md P5-7 / docs/specs/P5-2_実績からの予定提案.md(本仕様はP5-2のスコープ外欄 145-146行の拡張)
- 依存: P5-2(`computeSuggestions` / `PlanSuggestions`)、P5-1(定期ルール `createRecurringRuleAction`、`pattern` = daily/weekly/weekdays、weekly は複数曜日配列可)、P2-5(`createAppEventAction`)
- 指示資料: なし(`docs/指示資料/` を確認済み、2026-07-23時点でREADMEのみ)
- 経緯: 2026-07-23ユーザー質問「繰り返し(月〜金)のような提案はされるか」から発生。現状は
  タイトル×単一曜日でしか提案せず、平日毎日の作業が個別カードに割れて最大3件に埋もれる。
  方針・閾値・操作ボタン・「観測曜日のみで定期化」をユーザーと確定(2026-07-23)。

## 目的

同一タイトルが複数曜日で近い時間帯に繰り返されている実績を、**1枚のまとめカード**として提案する。
「平日毎日」「毎日」「特定の複数曜日(月・水・金など)」の繰り返しを、ワンアクションで
正しい定期予定(`weekdays` / `daily` / 複数曜日 `weekly`)として計画に取り込めるようにする。
DB変更なし・純粋関数(`computeSuggestions`)の拡張と既存Server Actionの再利用のみ。

## ユーザー確定事項(2026-07-23)

- **束ねる閾値**: 同一タイトルで開始時刻が互いに60分以内に揃う曜日が **3曜日以上** あれば1枚に束ねる
  (各曜日は既存どおり「直近4週で発生2日以上」を満たすものが対象)。2曜日以下は現行どおり個別カード
- **daily 対応**: 束ねた曜日集合が全7曜日なら `pattern: "daily"`
- **観測した曜日のみ定期化**: 「毎週にする」は観測された曜日だけを定期化する(勝手に未観測曜日を足さない)
  - 束ね集合が **全7曜日** → `pattern: "daily"`(ラベル「毎日」)
  - 束ね集合が **ちょうど月〜金** → `pattern: "weekdays"`(ラベル「毎週平日」)
  - それ以外(月・水・金など) → `pattern: "weekly"`, `weekdays: 観測曜日配列`(ラベル「毎週◯・◯・◯曜」)
- **操作ボタン**: まとめカードも「この週に追加」「毎週にする」の両方を出す

## 仕様

### 型の変更(`lib/calendar/suggestions.ts` の `PlanSuggestion`)

単一曜日前提の `weekday: number` / `date: string` を、複数曜日を表せる形に置き換える:

```ts
export interface PlanSuggestion {
  key: string;                              // "タイトル|pattern|曜日群(昇順join)" で一意化
  title: string;
  pattern: "weekly" | "weekdays" | "daily"; // 追加
  weekdays: number[];                       // 対象曜日(昇順。weekly単発は1要素)
  dates: string[];                          // 表示週内の対象各日 "YYYY-MM-DD"(weekdaysと同順)
  startTime: string;                        // "HH:mm"(ローカルTZ)
  endTime: string;                          // "HH:mm"(ローカルTZ)
  occurrenceCount: number;                  // 対象曜日クラスタの発生日数合計
}
```

- 単一曜日の提案は `pattern: "weekly"`, `weekdays: [w]`, `dates: [d]` として表現する(現行と等価)。

### 検出ロジック(`computeSuggestions` の拡張)

前段(タイトル×曜日のクラスタリング)は現行のまま。**後段に束ね処理を追加**する。

1. **曜日候補の生成(現行踏襲)**: タイトル×曜日でグループ化し、開始時刻60分幅のスライディング
   ウィンドウで発生日数最大のクラスタを取る。発生日数 < 2 の曜日は候補から外す。各曜日候補は
   「クラスタ開始時刻の中央値(束ね判定用)」と「クラスタ内の全実績(開始分・所要分)」を保持する。

2. **タイトル単位の束ね**: そのタイトルの曜日候補を中央値開始時刻で昇順に並べ、
   幅60分以内(差 ≤ 60)の**最大の連続ウィンドウ**を取る。
   - ウィンドウのサイズが **3以上** なら、その曜日集合 S を1つの束ね候補にする。
     S を取り除いた残りの曜日候補について同様に繰り返す(午前群・午後群など複数の束ねを許す)。
   - 残った(どの束ねにも入らない)曜日候補は、それぞれ単一曜日の候補にする。
   - ウィンドウ最大サイズが3未満なら、その時点の残り曜日候補はすべて単一曜日の候補。

3. **各候補の日時算出(束ね/単一 共通)**:
   - 対象曜日クラスタの**全実績の union** から、開始時刻 = 中央値を15分丸め、
     所要 = 中央値を15分丸め(最低15分)。終了 = 開始 + 所要。
   - 終了が24:00(1440分)を超える候補は**丸ごと除外**(FR-12の定期予定が日またぎ不可)。
   - 単一曜日の場合、union はその1曜日のクラスタなので算出結果は現行と等価(回帰なし)。

4. **対象各日の確定と除外(日単位)**: 候補の各曜日について、表示週の該当日
   (週は月曜始まり、`weekStart + (weekday+6)%7`)の提案日時を作り、次のいずれかなら**その曜日を除外**:
   - 提案開始日時が現在時刻より前(今週の過ぎた曜日。未来週は全曜日対象)
   - 表示週の同日に同タイトル(trim一致)の予定(`synced_events`)が既にある
   - 同タイトルの定期ルールがその曜日を覆う(`ruleCoversWeekday`: daily=全曜日 / weekdays=月〜金 / weekly=weekdays配列)
   - 残った曜日が**0**なら候補ごと非表示。**再分割はしない**(束ねが除外で2曜日以下に減っても
     1枚のまま。最終的な曜日集合から pattern を判定する)。

5. **pattern の最終判定**(残った曜日集合 W から):
   - W が全7曜日 → `daily`
   - W がちょうど {月,火,水,木,金}={1,2,3,4,5} → `weekdays`
   - それ以外 → `weekly`(`weekdays = W`)

6. **occurrenceCount**: 残った曜日の発生日数の合計。

7. **順序と上限(現行踏襲)**: 発生日数合計の降順 → 提案日時(残り日の最早)の昇順 →
   タイトルの昇順。上限は全体で最大3件(束ねカードは1件として数える)。

### 文言(`lib/calendar/messages.ts`)

`suggestionDescription` を「ラベル文字列」を受け取る形に変更し、pattern別ラベルを新設する:

```ts
suggestionPatternLabel: (pattern, weekdays) =>
  pattern === "daily"    ? "毎日"
  : pattern === "weekdays" ? "毎週平日"
  : `毎週${weekdays.map((w) => weekdayLabels[w]).join("・")}曜`,   // "毎週火曜" / "毎週月・水・金曜"
suggestionDescription: (label, startTime, durationLabel, count) =>
  `${label} ${startTime}頃・約${durationLabel}(直近4週で${count}回)`,
```

- 単一曜日weeklyの出力は現行と完全一致(例:「毎週火曜 09:00頃・約30分(直近4週で2回)」)。

### 受け入れ・却下(`components/plan-suggestions.tsx`)

- **この週に追加**: `dates` の各日について `createAppEventAction({ title, startAt, endAt })` を呼ぶ
  (`toUtcIso` でローカルTZ→UTC ISO変換。既存関数再利用)。
  - **部分失敗対応**: 成功があれば `router.refresh()`(成功分はカレンダーに反映され、次回再計算で
    その日は除外される)。1件でも失敗があればカードは残し `suggestionAddError` を表示。全成功なら
    カードを非表示。処理中は多重タップ防止で disabled。
- **毎週にする**: pattern別に `createRecurringRuleAction` を呼ぶ(既存Action。`recurring.ts` が
  `weekdays`/`daily` を `weekdays: null` で受理、`weekly` は複数曜日配列を受理):
  - `weekly`   → `{ pattern:"weekly",   weekdays: suggestion.weekdays }`
  - `weekdays` → `{ pattern:"weekdays", weekdays: null }`
  - `daily`    → `{ pattern:"daily",    weekdays: null }`
  - 共通: `title` / `startTime` / `endTime` / `timezone`(ブラウザTZ)/ `startsOn`(dates の最早日)/ `endsOn: null`
  - 成功でカード非表示 + `router.refresh()`(除外条件で再計算後も出ない)
- **×(非表示)**: 現行どおり `key` を useState で非表示(Action呼び出し・永続化なし)

### UI(ui-quality Skill / 375px)

- 現行の「実績からの提案」セクションをそのまま使う。候補0件・過去週・取得失敗時はセクションごと非表示。
- カードの説明文が pattern に応じて「毎日 …」「毎週平日 …」「毎週月・水・金曜 …」「毎週火曜 …」に変わる以外、
  レイアウト・色(群青系 `plan` トークン)・ボタンは現行踏襲。375pxで折り返しても押しやすいこと。

### エラー時の挙動

| 状況 | 挙動 |
|---|---|
| 過去実績の取得失敗(サーバー) | 現行どおりセクション非表示。カレンダー本体は通常描画 |
| 「この週に追加」全失敗 | カード内に日本語エラー、カードは残り再試行できる |
| 「この週に追加」一部失敗 | 成功分は反映(refresh)、失敗ぶんはカードに残りエラー表示 |
| 「毎週にする」失敗 | カード内に日本語エラー、カードは残る |

### セキュリティ

- 新しいデータ経路なし。読み取りは既存RLS(本人の行のみ)、書き込みは既存Server Action(認証確認済み)。
- 提案はクライアント計算のみ・DB保存なし。新規テーブル/ポリシーなし。

## スコープ外

- 却下の永続化(「二度と提案しない」)
- 見積もり係数(予定vs実績倍率)の学習・補正
- 時間帯不問の頻出タイトル提案 / 隔週・月次などの複雑な周期
- サマリー画面・計測画面への提案表示
- 束ね後に除外で曜日が減った際の再分割(1枚のまま最終曜日集合で表示する方針で確定)
