## 仕様

### A. エラー境界・404の日本語化

1. `app/not-found.tsx`(新規): 存在しないURLで表示。
   - 見出し「ページが見つかりません」+説明文+「トップへ戻る」リンク(`/`)
   - marketing/appどちらの配下でも破綻しない最小レイアウト(中央寄せ・トークン準拠)
2. `app/error.tsx`(新規): ルート配下の未捕捉例外で表示。
   - 見出し「問題が発生しました」+説明文+「再試行」ボタン(`reset()` を呼ぶ)
   - エラー詳細(スタック等)は表示しない(トークン・内部情報の漏洩防止)
3. `app/global-error.tsx`(新規): ルートレイアウト自体の例外で表示。
   - 自前の `<html lang="ja">`/`<body>` を含む自己完結マークアップ+「再読み込み」ボタン
4. 文言は `lib/errors/messages.ts`(新規)に集約する(既存の messages.ts パターンに準拠)

### B. 体感速度 — 遷移中のスケルトン表示

1. `app/(app)/loading.tsx`(新規): 認証必須4画面(calendar / track / summary / settings)への
   遷移中に即時表示されるスケルトン。
   - アプリシェル(AppBar・下部タブ)はlayoutが保持するため、コンテンツ領域のみのスケルトンとする
   - `aria-busy` 等で読み上げ環境にも配慮。文言が必要な場合は日本語
2. 実装前に `node_modules/next/dist/docs/` の該当ガイドを確認し、Next.js 16での
   loading境界の仕様(配置・適用範囲)に従う(AGENTS.md)
3. marketing側は静的ページのため対象外

### C. エラーメッセージ日本語化の総点検

- 全 `lib/*/messages.ts`・UIコンポーネント・APIルートを走査し、ユーザーの目に触れる
  英語文言・生のエラーコード表示が残っていないか確認する(調査済み分は上表の通り)
- 発見した場合は日本語化して本仕様書に追記する。**発見ゼロでもその旨をログに記録する**

### D. 375px幅の総点検

- 対象全13画面: `/`(LP)・`/pricing`・`/privacy`・`/terms`・`/login`・`/signup`・
  `/forgot-password`・`/reset-password`・`/calendar`・`/track`・`/summary`・`/settings`・
  `/onboarding` + 本タスクで追加する404/エラー画面
- Playwrightで 375px×ライト/ダークを走査し、横スクロール発生・はみ出し・タップ領域
  不足(44px未満)を検出する。主要画面は1280pxも確認
- 検出した崩れは本タスク内で修正する。修正が大きくなる場合(レイアウト変更を伴う等)は
  ユーザーに確認してから対応する

### E. デザイントークン化漏れの修正(総点検で発見。2026-07-13ユーザー承認で追加)

D(375px総点検)のダーク確認で、認証系4画面(login / signup / forgot-password /
reset-password)と規約系・マーケ共通部・設定の一部(計11ファイル)がD-1のトークン化から
漏れ、`zinc/red/emerald` 直書き+`dark:` バリアントのままであることを発見した。
`dark:` はOS設定にしか追随しないため、設定画面の手動テーマ(D-1e)でダークを選ぶと
背景(トークン)だけ暗転し、ボタン・入力欄・通知ボックスがライトのまま残る。

1. 対象11ファイルの直書きカラーをデザイントークン(`ink` / `ink-muted` / `line` /
   `surface` / `brand` / `danger` 等)へ置換し、`dark:` バリアントを全廃する
2. 認証系の主ボタンはLPのCTA規約(`bg-brand text-brand-ink … min-h-12 rounded-xl`)に揃える
3. 成功通知(メール送信完了等)用に `success` トークンを新設する(ライト/ダーク両対応。
   通知ボックスは `bg-success/10 text-success` パターン)
4. 認証系の補助リンク(高さ20px)と計測画面のヒントリンク(16px)はタップ領域44px以上へ
   拡大する(Dの検出事項。LP本文中のインラインリンクは文中リンクのため対象外)

## テストシナリオ

結合レベル(Route Handler / Server Action〜DB)のシナリオは含めない。
**理由**: 本タスクはDB・Route Handler・Server Actionに一切変更を加えず、追加するのは
静的なUI境界コンポーネント(error/not-found/loading)のみで、跨ぐ流れが存在しないため。
画面を跨ぐ挙動(404遷移・遷移中スケルトン)は手動シナリオM1〜M3で実機確認する。

- S1 [単体]: Given `not-found.tsx` When 描画する Then 「ページが見つかりません」の見出しと `/` へのリンクが表示される
- S2 [単体]: Given `error.tsx` When エラーオブジェクトを渡して描画する Then 「問題が発生しました」が表示され、エラーの内部情報(message/stack)は表示されない
- S3 [単体]: Given `error.tsx` When 「再試行」ボタンをクリックする Then `reset` が1回呼ばれる
- S4 [単体]: Given `global-error.tsx` When 描画する Then 日本語の見出しと「再読み込み」ボタンが表示される
- S5 [単体]: Given `(app)/loading.tsx` When 描画する Then スケルトンが `aria-busy` 付きで表示され、英語文言を含まない
- S6 [単体]: Given `lib/errors/messages.ts` When 各文言を参照する Then すべて日本語である(境界コンポーネントが文言集約を参照していることの担保)
- S7 [単体]: Given `app/`・`components/` の全ソース When 直書きカラークラス(`zinc-` / `red-` / `emerald-` / `sky-` / `amber-` と `dark:` バリアント)を走査する Then 1件も存在しない(トークン化の回帰防止)
- M1 [手動]: 存在しないURL(例 `/nonexistent`)へアクセス → 日本語404が375px/ライト・ダークで崩れず表示され、「トップへ戻る」で `/` に遷移する
- M2 [手動]: 全13画面+404を 375px×ライト/ダークで走査 → 横スクロールなし・はみ出しなし(主要画面は1280pxも確認)。検出した崩れは修正して再確認する
- M3 [手動]: 低速ネットワーク(Playwrightのthrottle)で下部タブから各画面へ遷移 → 遷移中にスケルトンが表示され、無反応にならない
