# 要件定義書 — PlanDiff

**バージョン**: 0.6(実績からの予定提案 FR-13 を追加)
**最終更新**: 2026-07-14
**ステータス**: ドラフト

---

## 1. プロダクト概要

### 1.1 一言コンセプト

**「見積もりが当たるようになる、エンジニアのためのカレンダー × タイムトラッカー」**

Googleカレンダーの「予定(計画)」と、ワンタップで記録する「実績(現実)」を同じタイムライン上に重ねて表示し、計画と現実のギャップを可視化する。使い続けるほど自分の見積もり癖が定量的にわかり、工数見積もりの精度が上がるプロダクト。

### 1.2 解決する課題

- エンジニアは工数見積もりを職業的に求められる(スプリントプランニング、リリース日回答)が、見積もりを外し続けても「自分がどうズレるのか」を定量的に知る手段がない
- カレンダーアプリは「予定」しか持たず、タイムトラッカーは「実績」しか持たない。両者を突き合わせる手段がない
- 会議や割り込みで開発時間が削られている実感はあるが、上司・PMに示せる客観的なデータがない
- 既存タイムトラッカー(Toggl等)は記録が目的化しており、「記録した結果、見積もりがどう改善されるのか」への回答が弱い

### 1.3 提供価値(コアループ)

```
計画する → 実行・記録する → ギャップを見る → 次の計画が賢くなる
   ↑                                              │
   └──────────────────────────────────────────────┘
```

1. **計画**: PlanDiff上で予定を作る(またはGoogleカレンダーの予定を同期する。連携は任意で、
   2026-07-10時点ではUI凍結中。詳細はFR-02/FR-10)
2. **記録**: 予定ブロックをタップしてタイマー開始/停止
3. **ギャップ可視化**: 予定と実績が同一タイムラインに重なって表示される
4. **学習**: 「計画◯時間 vs 実績◯時間、ズレ+◯%」のサマリーで自分の癖を知る

MVPでは1〜3を完成させ、4はサマリー表示まで(係数学習・サジェストはv2)。

## 2. ターゲット

### 2.1 メインターゲット

**ソフトウェアエンジニア個人**。まずは日本語圏のWeb系エンジニアをビーチヘッドとし、確立後にデザイナー・PMなど隣接職種、さらに一般の生産性向上層へ段階的に拡張する。

### 2.2 想定ペルソナ(初期)

- 20〜30代のWeb系エンジニア。Googleカレンダーで予定管理し、スプリントプランニングや工数見積もりを日常的に行う
- 見積もりを外して残業・信頼低下につながった経験があり、精度を上げたいと思っている
- 会議・割り込みで開発時間が細切れになっている実感があるが、定量的に示せていない
- Togglなどを試したが「記録するだけ」で続かなかった経験がある
- 開発者自身がこのペルソナに該当し、ドッグフーディングで高速に仮説検証する

### 2.3 非ターゲット(MVP時点)

- チーム・組織での工数管理(将来のTeam展開候補)
- 請求書発行が主目的のフリーランス(v2以降の拡張候補)
- 一般の生産性向上層(エンジニアで実証後の拡張先として保持)

## 3. ポジショニングと競合

| 競合 | 強み | 本プロダクトとの差分 |
|---|---|---|
| Toggl Track | 記録の手軽さ、レポート | 予定(計画)との突き合わせがない |
| Google カレンダー | 予定管理のデファクト | 実績の概念がない |
| Rize / Timely | 自動トラッキング | 自動記録が主で「計画とのギャップ」が主役ではない |
| Sunsama / Akiflow | デイリープランニング | タスク計画が主で、実績計測・ギャップ分析が弱い |
| WakaTime | IDE内コーディング時間の自動計測 | 「書いた時間」の記録であり、予定(計画)との突き合わせがない。競合ではなく将来のデータソース連携候補 |

**差別化の核**: 「予定 × 実績のオーバーレイ表示」と「見積もりズレの定量化」を体験の中心に据えること。機能の広さでは戦わない。

## 5. 機能要件詳細

### FR-01 Google認証

- Supabase AuthのGoogleプロバイダを使用
- 初回ログイン時にカレンダー読み取りスコープを同時リクエスト(`access_type=offline`、`prompt=consent`)
- `provider_refresh_token` を確実に取得・保存すること(取得できないケースをWeek 1で疎通確認)

### FR-02 カレンダー読み取り連携

- 対象: ユーザーのプライマリカレンダーのみ(MVP)
- 同期方式: 画面ロード時 + 手動リフレッシュのオンデマンド取得(push通知/webhookはMVPでは不使用)
- 取得範囲: 表示中の週 ± 1週間
- 取得した予定は `synced_events` テーブルにキャッシュ(APIコール削減とオフライン耐性)
- アクセストークン失効時はrefresh tokenで自動更新。refresh token失効時は再認可フローへ誘導
- **2026-07-10より当面凍結**: 環境フラグ(`GOOGLE_INTEGRATION_ENABLED`、既定OFF)でUI導線・同期APIを無効化。コード・テストは保持し、フラグONで復活する(P2-5)

### FR-10 予定のアプリ内作成・編集・削除(2026-07-10追加)

- タイトル・開始/終了日時のシンプルな予定をカレンダー画面から作成・編集・削除できる
- アプリ予定は `synced_events` に `source='app'`・`google_event_id='app:<uuid>'` で保存し、
  タイマー(FR-04)・オーバーレイ(FR-06)・サマリー(FR-07)は既存の紐づけキーのまま機能する
- Google由来の予定(`source='google'`)は従来どおり読み取り専用。同期処理はアプリ予定を削除しない
- 終日予定・D&D移動はスコープ外(v2以降)。繰り返し予定はFR-12参照

### FR-11 計測画面(2026-07-11追加)

- ルート `/track`(ナビ名「計測」)。カレンダーを経由せずタイマーの開始/停止と当日実績の確認ができる
- 実行中タイマーの表示・停止、フリータイマー開始(FR-05と同一仕様)、進行中・直近の予定からの
  予定連動開始(FR-04と同一仕様)を1画面に集約する
- 当日(ユーザーTZ)の確定済み実績を一覧表示する。編集・削除はカレンダー画面(FR-08)に委ねる
- **予定への昇格導線**: フリー実績から「次回の予定」(FR-10のアプリ内予定)をワンタップで
  作成できる。「記録が目的化」せずコアループ(§1.3)へ還流させるための必須要素

### FR-12 定期予定(繰り返し予定)(2026-07-14追加)

- FR-10(アプリ内予定)の拡張。パターンは**毎日 / 毎週(曜日複数選択)/ 平日(月〜金)**の3種
- 繰り返しルール(`recurring_rules`)を保存し、表示範囲の読み込み時にオンデマンドで
  `synced_events`(`source='app'`)へ実体化する。既存のタイマー(FR-04)・オーバーレイ(FR-06)・
  ギャップサマリー(FR-07)は無改修で機能する
- 編集・削除は「**この回のみ**」「**繰り返し全体**」の2択(「これ以降」は提供しない)。
  全体削除・全体編集でも**過去に発生した回のインスタンスは残す**(計画していた事実を保持)
- 期限は**無期限がデフォルト**。任意で終了日を指定可能
- 日をまたぐ繰り返し(終了時刻が開始時刻より前)はスコープ外
- 詳細は `docs/specs/P5-1_定期予定.md` を参照

### FR-13 実績からの予定提案(2026-07-14追加)

- カレンダー画面に「実績からの提案」セクションを常設。表示週の直前4週間の完了実績から
  「同タイトル × 同曜日 × 開始時刻の差が60分以内」で2日以上発生した繰り返しを検出し、
  過去実績の中央値(15分単位丸め)の開始時刻・所要時間で日時入りの提案を出す(最大3件)
- 受け入れは「**この週に追加**」(FR-10の単発予定)/「**毎週にする**」(FR-12の定期予定)の2択
- 表示週に同タイトルの予定・該当曜日を含む同タイトルの定期ルールが既にある場合、
  提案日時が過去の場合は提案しない。過去週の表示中は提案しない
- 却下(×)はその場限りの非表示。永続化しない(提案データはDBに保存しない)
- 詳細は `docs/specs/P5-2_実績からの予定提案.md` を参照

### FR-04/05 タイマー

- **実行中タイマーは常に1本**(サーバー側で管理)。新規開始時は既存タイマーを自動停止
- タイマー状態はDBに永続化し、PWA/複数デバイス間でリロード後も復元可能
- 予定連動タイマーは `google_event_id` で予定と紐づける
- 停止時に実績エントリ(`time_entries`)として確定

### FR-06 オーバーレイ表示

- 予定ブロック: 左寄せ・薄い塗り
- 実績ブロック: 右寄せ・濃い塗り(予定と紐づく場合は同系色)
- 予定と実績のズレ(開始遅延・超過時間)が視覚的にわかること
- **この画面がプロダクトの「顔」。UI品質に最も投資する**

### FR-07 ギャップサマリー

- 今日/今週の集計: 計画合計、実績合計、ズレ(絶対値と%)
- 予定単位の一覧: 予定名、計画時間、実績時間、ズレ
- 実績が紐づかない予定(未着手)、予定に紐づかない実績(割り込み作業)も区別して表示

## 7. 非機能要件

### 7.1 セキュリティ

- 全テーブルにRLSを適用。ヘルパー関数は `private` スキーマに配置し `SECURITY DEFINER` で実装(家計簿アプリで確立したパターンを踏襲)
- Googleのrefresh tokenは平文でクライアントに渡さない。サーバーサイド(Route Handler / Edge Function)でのみ使用
- トークン類はSupabase上で暗号化保存を検討(最低限、専用テーブル + RLSでユーザー本人以外アクセス不可)

### 7.2 プライバシー / コンプライアンス(OAuth審査の必須要件)

- プライバシーポリシーの公開(審査必須。カレンダーデータの用途・保存範囲・削除方法を明記)
- Google API Services User Data Policy の **Limited Use 要件**への準拠を明記(カレンダーデータを広告・第三者提供・本機能以外に使わない)
- ユーザーによるデータ全削除機能(設定画面から)

### 7.3 パフォーマンス

- カレンダー画面の初期表示: キャッシュ表示を先行し、同期は非同期で反映
- タイマー開始/停止の体感遅延: 楽観的UI更新で即時反映

### 7.4 対応環境

- モバイル: iOS Safari / Android Chrome(PWA)
- デスクトップ: Chrome / Safari / Edge 最新版
- レスポンシブ必須(モバイルファーストで設計)

## 8. データモデル概要

```
profiles          … ユーザープロフィール(auth.usersと1:1)
google_tokens     … provider refresh token等の保管(サーバーのみアクセス)
synced_events     … 予定(Googleカレンダーのキャッシュ+アプリ内作成の予定)
                    (source 'google'|'app', google_event_id, title,
                     start_at, end_at, synced_at)
                    アプリ予定は google_event_id='app:<uuid>'(2026-07-10改定)
time_entries      … 実績記録
                    (title, start_at, end_at, google_event_id nullable,
                     end_at が null = 実行中タイマー)
```

- `time_entries.google_event_id` が予定と実績を結ぶキー
- 「実行中タイマーは1本」制約: `end_at IS NULL` の行がユーザーごとに最大1件(partial unique index)

## 12. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| OAuth審査が1ヶ月内に完了しない | 一般公開の遅延 | Week 1で申請着手。未審査中はテストユーザー(上限あり)でベータ運用し、公開はソフトローンチに切り替え |
| refresh token取得のハマり | 連携機能全体が止まる | Week 1最優先で疎通確認。Supabase Authのクエリパラメータ設定を最初に検証 |
| 「記録が続かない」既存トラッカーと同じ離脱 | 継続率低下 | 記録ゼロでも「予定だけで使える」カレンダーとして成立させ、タップ1回で記録できる導線に絞る |
| エンジニア特化で市場規模が小さくなる | 成長上限 | ビーチヘッド戦略と割り切る。継続率・課金意欲を実証後、デザイナー・PM・一般層へ段階拡張 |
| WakaTime等と混同され「自動計測ツール」と誤解される | 訴求のズレ | LPで「予定 vs 実績のギャップ」を主語にし、コーディング計測ツールではないことを明確化 |
